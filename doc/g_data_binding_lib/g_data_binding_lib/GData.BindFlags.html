BindFlags

Description:

[ Flags ]
public enum BindFlags

Flags specified upon creation of new BindingInterface to describe how binding should act

Since:

0.1


Namespace: GData
Package: g_data_binding_lib

Content:

Enum values:

  • DEFAULT - Default is simply automated transferal from source to target without initial syncing
  • SYNC_CREATE - Specifies that data should be synced upon creation from source to target. This direction is reversed if there is also specified flag REVERSE_DIRECTION.
  • BIDIRECTIONAL - Specifies data is transfered both ways.
  • CONDITIONAL_WRITE - Specifies write is not necessarily possible. If this flag is set then BIDIRECTIONAL will be disabled if that is so
  • CONDITIONAL_BIDIRECTIONAL - Simple combination of BIDIRECTIONAL and CONDITIONAL_WRITE for easier use
  • INVERT_BOOLEAN - Specifies boolean inversion when assigning boolean value.
  • REVERSE_DIRECTION - Controls direction of SYNC_CREATE and data transfer, default direction is from source to target and this flag reverses that.
  • MANUAL_EVENTS_FROM_SOURCE - Specifies that binding should not connect to source events as they will be provided manually by triggering update_from_source()
  • MANUAL_EVENTS_FROM_TARGET - Specifies that binding should not connect to target events as they will be provided manually by triggering update_from_target()
  • MANUAL_UPDATE - This is equal to specified both MANUAL_EVENTS_FROM_SOURCE and MANUAL_EVENTS_FROM_TARGET
  • INACTIVE - Specifies active status and is update by freeze()/unfreeze() Binding created with this flag must call unfreeze manually
  • SOURCE_UPDATE_FLOOD_DETECTION - Specifies that data transfer from target should be checked for data flooding
  • TARGET_UPDATE_FLOOD_DETECTION - Specifies that data transfer from source should be checked for data flooding
  • FLOOD_DETECTION - FLOOD_DETECTION is equal to specifying both SOURCE_UPDATE_FLOOD_DETECTION and TARGET_UPDATE_FLOOD_DETECTION
  • SOURCE_DELAYED - Similar to flood detection this provides static delay to data transfer if another event occurs during delay, delay is prolonged for another delay_interval. example usecase is binding search controls where you don't want to spam search requests
  • TARGET_DELAYED - Similar to flood detection this provides static delay to data transfer if another event occurs during delay, delay is prolonged for another delay_interval. example usecase is binding search controls where you don't want to spam search requests
  • DELAYED - Similar to flood detection this provides static delay to data transfer if another event occurs during delay, delay is prolonged for another delay_interval. example usecase is binding search controls where you don't want to spam search requests
  • CUSTOM_EVENTS_ONLY - Specifies binding data transfer only relies on additionally specified events and original property should not be connected
  • SUPPRESS_WARNINGS - Specifies all error reporting should be suppressed as there is no guarantee that binding is possible
  • USE_ORIGINAL_SOURCE_TYPE - Specifies binding should not use registered handler and instead use handler specified in its original_type property
  • USE_ORIGINAL_TARGET_TYPE - Specifies binding should not use registered handler and instead use handler specified in its original_type property
  • USE_ORIGINAL_TYPES - Specifies binding should not use registered handler and instead use handler specified in its original_type property
  • DEBUG - Specifies debugging is enabled for binding
